User

    get /users
        - returns all active users, based on who is requesting, the response will be filtered, i.e., the franchise_staff can only read self info, franchise_admin can read self and self franchise staff info, main_admin can read every users info
        - returns info like : id, username, role, franchise_id, full_name, email, extra_permission, revoked_permission
    get /users/{user_id}
        - returns info of specific user based on user_id, all rules from above endpoint applies here also
    get /users?search={value}&full_name={full_name}&username={username}&franchise_id={franchise_id}&email={email}&role={role}
        - returns info of specific user based on query parameters, all rules from above endpoint applies here also
    post /users
        - based on role, not evey user will be able to create every type of user
        - the scope in which user will be created and type of that users will be limited to few user types
        - extra permission and revoked permission must be handled with care, not every user will be eligible for those
        - if main_admin is creating the user then franchise_id must be provided
        - if franchise_admin is creating the user then franchise_id is optional and any provided franchise_id other than its own franchise is ignored
        - franchise_admin can create franchise_admin and franchise_staff_member in its own franchise
        - request body : {username, password, role, franchise_id?, full_name, email?, extra_permissions, revoked_permissions }
        - the password will stored in one way increapted format in db
        - return json : { id, username, role, franchise_id, full_name, email?, created_at, updated_at }
    patch /users/{user_id}/profile
        - can only update the profile info like full_name, email
        - can still be updated by users with higher proper clearance
        - request_body : { full_name, email }
        - updates the user's profile info
        - returns json : { user_id, updated_at }
    patch /users/{user_id}/access
        - can only update the access related info like franchise_id, role
        - only possible by main_admin
        - request_body : { franchise_id, role }
        - returns json : { user_id, updated_at }
    patch /users/{user_id}/permissions
        - can only adjust permission overrides on a user; only main_admin
        - request_body : { extra_permissions, revoked_permissions }  **both arrays are deltas, not full replacement**
        - server merges each delta into the user row using role defaults + current extra_permissions + current revoked_permissions
        - extra_permissions (delta): for each permission code in this array:
            - if it is currently in revoked_permissions and is part of this user's role default set, remove it from revoked_permissions (restore that default grant)
            - else if it is not part of the role default set, add it to extra_permissions if not already present
            - if it is already granted by defaults and not revoked, no change
        - revoked_permissions (delta): for each permission code in this array:
            - if it is in extra_permissions, remove it from extra_permissions
            - else if it is part of the role default set, add it to revoked_permissions (revoke that default grant)
            - if the same code appears in both delta arrays in one request, the call is rejected
            - if a revoke delta targets a permission that is neither in role defaults nor in extra_permissions, that entry is ignored and may be logged (invalid / stale UI state)
        - returns json : { user_id, updated_at }
        - for the full current state and UI hints, use get /users/{user_id}/permissions
    get /users/permissions?role={role}
        - only main_admin can access this endpoint
        - role can only be franchise_admin or franchise_staff_member
        - returns the permission information needed while creating or updating user permission overrides
        - returns json : {
            role,
            default_permissions,
            extra_permissions_available,
            revokable_permissions
          }
    get /users/{user_id}/permissions
        - only main_admin can access this endpoint
        - returns the current permission configuration of the existing user
        - returns json : {
            user_id,
            role,
            default_permissions,
            extra_permissions,
            revoked_permissions,
            effective_permissions,
            extra_permissions_available,
            revokable_permissions,
            removable_extra_permissions
          }
    patch /users/{user_id}/deactivate
        - based on role, not every user will be able to perform this operation
        - main_admin can deactiave the franchise_admin and franchise_staff, the franchise_admin can only deactive its own franchise's staff
        - request_body : {}
        - deactivates the user
        - returns json : { id, is_activate, updated_at }
    patch /users/{user_id}/activate
        - based on role, not every user will be able to perform this operation
        - main_admin can activate the franchise_admin and franchise_staff, the franchise_admin can only activate its own franchise's staff
        - request_body : {}
        - activates the user
        - returns json : { id, is_activate, updated_at }
    patch /users/{user_id}/reset-password
        - based on role, not every user can reset password of other user
        - main_admin can reset password of any lower-level user
        - franchise_admin can reset password of its own franchise staff
        - this endpoint is only for admin-driven password reset of another user
        - request_body : { new_password }
        - returns json : { user_id, updated_at }

        - very important **
        - franchise_admin & staff can only create booking for their franchise, main_admin can create booking for any franchise
        - the request body management is crucial
        - in request body customer_id can be provided if not, the actual customer info like full_name, mobile, email? must be provided
        - in request body vehicle_id can be provided if not, the actual vehicle info like name?, customer_id, franchise_id, registration_number, vehcile_category must be provided

        - ** for now, the seed data must be present in the database like : customer, vehicle, service
        - request body : { franchise_id, customer_id | customer_info, vehicle_id | vehicle_info, requested_at, notes?, requested_services : [{ service_id, qty }, ...], gst_included } — created_by is the authenticated user (not sent in body)
        - returns json : 
            {
                id, franchise_id, customer_id, vehicle_id, requested_at, created_by, notes, created_at, updated_at,
                booking_items : [{
                    ...,
                    service : {...}
                }],
                vehicle : {...},

                customer : {...},
                invoice : {
                    ...,
                    gst_included
                    payments : [

        - very important **

        - input for this endpoint is very specific : [{service1, qty1}, {service2, qty2}, ...]
        - so we can increase/decrease the qty of service taken, decreasing the qty to 0 must remove that service from service mapped to that booking
        - for services which are coming from request body, if those are not mapped then we map if, for services which are already mapped we modify qty for those
        - after the service config mapping is done, we modify the invoice details like, total_base_amount, total_payable_amount, and payment_status if total_paid_amount becomes >= new_total_payable_amount
        - request body : { items : [{service1, qty1}, {service2, qty2}, ...] }
        - response body : { booking_id, updated_at }

        - based on who is requesting this info, not every one can see all payments
        - franchise_admin and franchise_staff can only see payments under their franchise, main_admin can see all payments
        - return json : { id, invoice_id, amount, mode, verified_by, reference_number }
    post /payments
        - very rare usage, never use it : must be handled via : post /invoices/{invoice_id}/payments
        - for complete ness sack, this endpoint can add 1 payment record in invoice
        - based on added payment, the total_paid_amount and payment_status can change in invoice
        - look for over payment **
        - request body : { invoice_id, amount, mode, verified_by, reference_number? }
        - returns json : { id, invoice_id, amount, mode, verified_by, reference_number, created_at, updated_at }
        - not provided in v1
    patch /payments/{payment_id}
        - only reference_number can be updated rest of the things are immutable
        - franchise_admin and franchise_staff can only update payments under their franchise, main_admin can see update payments
        - request body : { reference_number }
        - returns json : { id, updated_at }
